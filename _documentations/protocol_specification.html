ALG 5.1 Voter Registration — Key Derivation & Credential Issuance
Input  NID number, password, face image
Output  QR credential (PNG), public key \(S\) committed to blockchain
Step 1 — Biometric Feature Extraction
  1. Detect face bounding box using MediaPipe BlazeFace; crop and resize to \(160 \times 160 \times 3\).
  2. Extract FaceNet embedding:
    \[\mathbf{e} = \text{FaceNet}(\text{face\_crop}) \in \mathbb{R}^{128}, \quad \|\mathbf{e}\|_2 = 1\]
  3. Apply PCA projection to 64 dimensions and re-normalise:
    \[\mathbf{e}' = \frac{(\mathbf{e} - \boldsymbol{\mu}) \cdot W}{\|(\mathbf{e} - \boldsymbol{\mu}) \cdot W\|_2} \in \mathbb{R}^{64}\]
    where \(\boldsymbol{\mu} \in \mathbb{R}^{128}\) is the training-set mean and \(W \in \mathbb{R}^{128 \times 64}\) is the top-64 PCA eigenvector matrix.
Step 2 — Deterministic Private Key Derivation
  1. Hash the normalised embedding:
    \[h_{\text{face}} = \text{SHA-256}(\mathbf{e}')\]
  2. Sample a cryptographic salt:
    \[\text{salt} \xleftarrow{\$} \{0,1\}^{128}\]
  3. Derive the private key by hashing the concatenation and reducing modulo the curve order \(n\):
    \[k = \text{SHA-256}(h_{\text{face}} \,\|\, \text{salt}) \bmod n \in [1,\, n-1]\]
  4. Compute the corresponding secp256k1 public key:
    \[S = k \cdot G\]
    where \(G\) is the secp256k1 generator point.
Step 3 — Blockchain Commitment
  1. Compute \(\text{nidHash} = \text{SHA-256}(\text{NID})\) to avoid storing raw ID on-chain.
  2. Submit transaction register(nidHash, S_x, S_y, salt) to Hyperledger Fabric chaincode.
  3. Chaincode appends \(S\) to the GLOBAL_RING key in the world state.
Step 4 — QR Credential Generation
  1. Derive AES key from the voter's password:
    \[K_{\text{aes}} = \text{SHA-256}(\text{password})\]
  2. Construct plaintext payload:
    \[P = \mathbf{e}' \,\|\, h_{\text{face}} \,\|\, \text{salt} \,\|\, \text{nidHash}\]
  3. Encrypt with AES-256-CBC and a random IV:
    \[\text{QR\_payload} = \text{IV} \,\|\, \text{AES-CBC}_{K_{\text{aes}}}(P)\]
  4. Encode payload as a QR code PNG and return to voter. The raw embedding and private key are never persisted anywhere.
⚠ Security Properties of Registration
The private key \(k\) exists only in transient server memory during one request and is never stored. The blockchain holds only \((S_x, S_y)\), which by the ECDLP reveals nothing about the face embedding. Without the correct password, the QR payload cannot be decrypted to recover \(\mathbf{e}'\) or reconstruct \(k\).
2Phase

Ballot Creation Protocol

Ballot initialisation and homomorphic encryption keypair generation

ALG 5.2 Ballot Creation
Input  Title \(T\), description \(D\), candidate options \(\{O_1,\ldots,O_m\}\), start time \(t_s\), end time \(t_e\)
Output  Active ballot record \(B\)
  1. Validate: \(m \geq 2\), all options distinct, \(t_s < t_e\).
  2. Construct ballot object \(B\) with \(B.\text{id} = \texttt{"ballot\_"} \|\, \text{timestamp}\), \(B.\text{status} = \texttt{"active"}\), and \(B.\text{options} = \{(O_i,\, \text{votes}{:}\, 0)\}_{i=1}^{m}\).
  3. If a previous ballot exists, close it and archive it to history.
  4. Set \(B\) as the active ballot.
ALG 5.3 Homomorphic Encryption Setup — Exponential ElGamal KeyGen
Input  Ballot ID
Output  Public key \(Q\) published; private key \(d\) held server-side
  1. Sample a random private key:
    \[d \xleftarrow{\$} [1,\, n-1]\]
  2. Compute the corresponding public key:
    \[Q = d \cdot G\]
  3. Store the keypair \((d,\, Q)\) keyed by \(B.\text{id}\).
  4. Publish \(Q\) via GET /api/v1/tally/publickey/:ballotId.
⚠ Production Note
In a production deployment, the private key \(d\) should be generated through a Distributed Key Generation (DKG) ceremony among multiple trustees so that no single party can unilaterally decrypt individual votes before the election concludes.
3Phase

Vote Casting Protocol

Anonymous vote submission via biometric authentication, key recovery, ring signature, and homomorphic encryption

ALG 5.4 Anonymous Vote Casting
Input  QR code image, live face image, password, vote choice \(v_c \in \{O_1,\ldots,O_m\}\)
Output  Confirmed vote record on blockchain
Phase 3a — Biometric Authentication
  1. Decode the QR image using jsQR to retrieve the encrypted payload.
  2. Recover the AES key and decrypt the payload:
    \[K_{\text{aes}} = \text{SHA-256}(\text{password})\] \[P = \text{AES-CBC-Dec}_{K_{\text{aes}}}(\text{QR\_payload})\]
    Extract \((\mathbf{e}'_{\text{reg}},\, h_{\text{face}},\, \text{salt},\, \text{nidHash})\) from \(P\).
  3. Capture a live face image and extract its FaceNet + PCA embedding \(\mathbf{e}'_{\text{live}}\) using the same pipeline as registration.
  4. Compute the cosine similarity between the registered and live embeddings:
    \[\cos\theta = \frac{\mathbf{e}'_{\text{reg}} \cdot \mathbf{e}'_{\text{live}}}{\|\mathbf{e}'_{\text{reg}}\|_2 \; \|\mathbf{e}'_{\text{live}}\|_2}\]
  5. Apply the acceptance decision rule with threshold \(\tau = 0.50\):
    \[\text{Authenticate} = \begin{cases} \text{ACCEPT} & \text{if } \cos\theta > \tau \\ \text{REJECT} & \text{otherwise} \end{cases}\]
Phase 3b — Deterministic Private Key Recovery
  1. Recompute the private key identically to registration:
    \[k = \text{SHA-256}(h_{\text{face}} \,\|\, \text{salt}) \bmod n\]
  2. Verify the recovered key against the registered public key:
    \[S = k \cdot G \quad \text{must match} \quad S_{\text{registered}}\]
Phase 3c — Linkable Ring Signature Generation
  1. Retrieve the global ring from the blockchain: \(\mathcal{R} = \{PK_1, PK_2, \ldots, PK_n\}\).
  2. Locate the signer index \(\pi\) such that \(PK_\pi = S\).
  3. Execute the Liu–Wei–Wong LRS signing algorithm (detailed in Algorithm 5.5 below):
    \[\sigma = \text{LRS.Sign}(k,\; \mathcal{R},\; \pi,\; v_c) = (c_0,\; \{s_i\}_{i=0}^{n-1},\; I,\; \text{startIndex})\]
    where:
    • \(c_0\) is the initial challenge scalar
    • \(\{s_i\}\) are the \(n\) response scalars
    • \(I = k \cdot H_p(PK_\pi)\) is the link tag (a curve point)
Phase 3d — Vote Encryption (Exponential ElGamal)
  1. Retrieve the ballot public key \(Q = d \cdot G\) from the API.
  2. Sample a fresh random encryption nonce:
    \[r \xleftarrow{\$} [1,\, n-1]\]
  3. Encrypt the vote value \(v = 1\) (one for the chosen candidate):
    \[C_1 = r \cdot G\] \[C_2 = 1 \cdot G + r \cdot Q = G + r \cdot Q\]
    The resulting ciphertext is \(\text{Enc}(1) = (C_1,\, C_2)\).
  4. Serialise: \(\text{encVote} = \{C_1\!: (x, y),\; C_2\!: (x, y)\}\).
Phase 3e — Blockchain Submission & On-Chain Verification
  1. Verify the signature locally: \(\text{LRS.Verify}(\sigma,\, \mathcal{R},\, v_c) = 1\).
  2. Submit to the Hyperledger Fabric chaincode: castVote(v_c, σ, R, encVote).
  3. The chaincode extracts the link tag \(I = (I_x, I_y)\) from \(\sigma\).
  4. Check: if key LINK_TAG_Ix_Iy already exists in the world state → REJECT with "Double voting detected".
  5. Atomically: increment VOTE_COUNTER, store vote record under VOTE_count, write LINK_TAG_Ix_Iy, and emit a VoteCast event.
ALG 5.5 Liu–Wei–Wong Linkable Ring Signature — Sign
Input  Signer secret key \(k\), ring \(\mathcal{R} = \{PK_0, \ldots, PK_{n-1}\}\), signer index \(\pi\), message \(m\)
Output  Signature \(\sigma = (c_0, \{s_i\}, I, \pi)\)
  1. Compute the link tag using the hash-to-point function \(H_p\):
    \[I = k \cdot H_p(PK_\pi)\] where \(H_p(P) = \text{SHA-256}(P_x \,\|\, P_y) \cdot G\)
  2. Sample a random scalar and compute the signer's commitments:
    \[u \xleftarrow{\$} [1,\, n-1]\] \[L_\pi = u \cdot G, \qquad R_\pi = u \cdot H_p(PK_\pi)\]
  3. Compute the initial challenge for the next index \((\pi+1) \bmod n\):
    \[c_{\pi+1} = \text{SHA-256}(m \,\|\, L_\pi \,\|\, R_\pi)\]
  4. For \(i = \pi+1, \, \pi+2, \ldots, \pi + n - 1\) (indices taken mod \(n\)):
    1. Sample \(s_i \xleftarrow{\$} [1,\, n-1]\).
    2. Compute ring members' commitments: \(L_i = s_i \cdot G + c_i \cdot PK_i\) and \(R_i = s_i \cdot H_p(PK_i) + c_i \cdot I\).
    3. Compute the next challenge: \(c_{i+1\bmod n} = \text{SHA-256}(m \,\|\, L_i \,\|\, R_i)\).
  5. Close the ring: compute the signer's response scalar:
    \[s_\pi = u - c_\pi \cdot k \pmod{n}\]
  6. Output \(\sigma = (c_0,\; \{s_i\}_{i=0}^{n-1},\; I,\; \pi)\).
4Phase

Revote Prevention — Double-Vote Detection

Cryptographic uniqueness guarantee via link tag determinism

Theorem 5.1 — Revote Prevention Correctness
If a voter with private key \(k\) produces two valid ring signatures \(\sigma_1\) and \(\sigma_2\) (over any rings containing \(PK = k \cdot G\)), their link tags satisfy:
\[I_1 = I_2\]
with probability 1. Consequently, the chaincode will reject the second vote upon detecting the duplicate link-tag key.
Proof
The link tag is defined as:
\[I = k \cdot H_p(PK)\]
where \(H_p\) is deterministic and \(PK = k \cdot G\) is uniquely determined by \(k\). Since \(k\), \(PK\), and \(H_p\) are all deterministic, the value \(I\) is uniquely a function of \(k\) alone:
\[I = f(k) = k \cdot H_p(k \cdot G)\]
Therefore \(I_1 = k \cdot H_p(PK) = I_2\) for any two signatures by the same signer.   \(\square\)
Theorem 5.2 — Privacy of Link Tags
Given a link tag \(I\) and the ring \(\mathcal{R} = \{PK_1, \ldots, PK_n\}\), no probabilistic polynomial-time (PPT) adversary can determine which ring member produced \(I\) with advantage non-negligibly greater than \(\frac{1}{n}\), under the ECDLP hardness assumption.
Proof Sketch
Identifying the signer from \(I\) and \(\mathcal{R}\) requires finding \(\pi\) such that \(I = k_\pi \cdot H_p(PK_\pi)\). Since \(H_p(PK_\pi)\) is a known public point \(\mathcal{P}_\pi\), this reduces to finding \(k_\pi\) satisfying:
\[I = k_\pi \cdot \mathcal{P}_\pi\]
which is exactly the Elliptic Curve Discrete Logarithm Problem (ECDLP) on the secp256k1 curve — assumed computationally intractable.   \(\square\)
5Phase

Vote Tallying Protocol

Privacy-preserving tally via homomorphic aggregation and Baby-Step Giant-Step decryption

ALG 5.6 Homomorphic Tally Computation
Input  All encrypted votes \(\{(v_{c_j}, \text{encVote}_j)\}_{j=1}^{N}\) from blockchain, keypair \((d, Q)\), candidates \(\{O_1,\ldots,O_m\}\)
Output  Vote counts \(\{\text{count}_i\}_{i=1}^{m}\)
Step 1 — Homomorphic Aggregation
  1. Retrieve all \(N\) encrypted vote records from the Hyperledger Fabric ledger.
  2. For each candidate \(O_i\), initialise the aggregate ciphertext \(\mathcal{C}_i = \bot\).
  3. For each vote \(j\) with \(v_{c_j} = O_i\), deserialise \(\text{ct}_j = \text{Enc}(1) = (C_{1,j},\, C_{2,j})\) and accumulate using the homomorphic addition operation \(\oplus\):
    \[\mathcal{C}_i = \bigoplus_{j:\, v_{c_j} = O_i} \text{ct}_j\]
    Concretely, for two ciphertexts \((C_1,\, C_2)\) and \((C_1',\, C_2')\):
    \[(C_1,\, C_2) \oplus (C_1',\, C_2') = (C_1 + C_1',\;\; C_2 + C_2')\]
    After aggregating \(\text{count}_i\) individual encryptions of 1, the result equals:
    \[\mathcal{C}_i = \text{Enc}(\text{count}_i) = \!\!\!\!\!\! \underbrace{\left(\sum_{j:\, v_{c_j}=O_i} r_j\right)}_{\tilde{r}} \!\!\!\! \cdot G,\;\; \text{count}_i \cdot G + \tilde{r} \cdot Q\bigg)\]
Step 2 — Exponential ElGamal Decryption
  1. For each candidate \(O_i\) with aggregate ciphertext \(\mathcal{C}_i = (\tilde{C}_1,\, \tilde{C}_2)\), decrypt to obtain the encoded vote total:
    \[M_i = \tilde{C}_2 - d \cdot \tilde{C}_1 = \text{count}_i \cdot G\]
    Correctness follows from:
    \[M_i = (\text{count}_i \cdot G + \tilde{r} \cdot Q) - d \cdot (\tilde{r} \cdot G) = \text{count}_i \cdot G + \tilde{r}(dG) - d(\tilde{r} G) = \text{count}_i \cdot G\]
Step 3 — Baby-Step Giant-Step Discrete Log Recovery

We hold \(M_i = \text{count}_i \cdot G\) and need to recover the integer \(\text{count}_i \in [0, N]\).

  1. Set the stride \(m = \lceil\sqrt{N}\rceil\).
  2. Baby Steps — build a lookup table:
    \[T = \{(j \cdot G,\; j) \mid j = 0, 1, \ldots, m-1\}\]
  3. Compute the giant-step increment \(\Delta = m \cdot G\).
  4. Giant Steps — set \(\gamma = M_i\) and iterate for \(i = 0, 1, \ldots, m\):
    1. If \(\gamma \in T\): return \(\text{count}_i = i \cdot m + T[\gamma]\).   ✓
    2. Update \(\gamma \leftarrow \gamma - \Delta\).
Complexity: Time \(\mathcal{O}(\sqrt{N})\), Space \(\mathcal{O}(\sqrt{N})\). For \(N \leq 10{,}000\) voters this requires at most \(\sqrt{10{,}000} = 100\) curve-point operations — effectively instantaneous.
ALG 5.7 Public Tally Verification
Input  All encrypted votes (public on blockchain), claimed aggregate ciphertexts \(\{\mathcal{C}_i\}\), claimed counts \(\{\text{count}_i\}\)
Output  Verification result \(\in \{\text{VALID}, \text{INVALID}\}\)
  1. Retrieve all encrypted votes from the public ledger independently.
  2. Recompute the homomorphic sums \(\mathcal{C}_i'\) for each candidate using the same aggregation procedure.
  3. Check equality of aggregate ciphertexts:
    \[\mathcal{C}_i' \stackrel{?}{=} \mathcal{C}_i \quad \forall\, i = 1, \ldots, m\]
    If all match, the aggregation step was performed faithfully.
Note: Verifying the decryption step — i.e. that \(\text{count}_i \cdot G = M_i\) was computed from the correct secret \(d\) — requires either access to \(d\) or a zero-knowledge proof of correct decryption such as the Chaum–Pedersen protocol.
6Phase

Vote Validity Proof — ZK Range Proof

Disjunctive Chaum–Pedersen protocol to prevent ballot stuffing

Goal: Prove that a ciphertext \((C_1, C_2)\) encrypts a value in \(\{0, 1\}\) without revealing which, so that a malicious voter cannot encrypt an arbitrary large integer to inflate a candidate's count.
ALG 5.8 1-of-2 Disjunctive Chaum–Pedersen ZKP — Prove Valid Vote
Input  Ciphertext \((C_1, C_2) = (rG,\; v \cdot G + rQ)\) with \(v \in \{0,1\}\), randomness \(r\), ballot public key \(Q\)
Output  Proof \(\pi = (a_1, b_1, a_2, b_2, d_1, d_2, r_1, r_2)\)
Case: Actual Vote is 0  (real proof for statement 0, simulated for statement 1)
  1. Sample random scalars:
    \[w,\, d_2,\, r_2 \xleftarrow{\$} [1,\, n-1]\]
  2. Compute the real commitment for statement 0 (\(v = 0\)):
    \[a_1 = w \cdot G, \qquad b_1 = w \cdot Q\]
  3. Compute the simulated commitment for statement 1 (\(v = 1\)):
    \[a_2 = r_2 \cdot G + d_2 \cdot C_1\] \[b_2 = r_2 \cdot Q + d_2 \cdot (C_2 - G)\]
  4. Compute the Fiat–Shamir challenge:
    \[c = \text{SHA-256}(C_1,\, C_2,\, a_1,\, b_1,\, a_2,\, b_2)\]
  5. Split the challenge to satisfy \(d_1 + d_2 \equiv c \pmod{n}\):
    \[d_1 = c - d_2 \pmod{n}\]
  6. Compute the real response:
    \[r_1 = w - d_1 \cdot r \pmod{n}\]
  7. Output proof \(\pi = (a_1,\, b_1,\, a_2,\, b_2,\, d_1,\, d_2,\, r_1,\, r_2)\).
ALG 5.9 1-of-2 Disjunctive Chaum–Pedersen ZKP — Verify
Input  Ciphertext \((C_1, C_2)\), proof \(\pi = (a_1, b_1, a_2, b_2, d_1, d_2, r_1, r_2)\), ballot public key \(Q\)
Output  \(\{0,1\}\)
#CheckEquation
1 Challenge consistency \[d_1 + d_2 \equiv \text{SHA-256}(C_1, C_2, a_1, b_1, a_2, b_2) \pmod{n}\]
2 Statement 0 commitment — \(G\)-component \[r_1 \cdot G + d_1 \cdot C_1 \stackrel{?}{=} a_1\]
3 Statement 0 commitment — \(Q\)-component \[r_1 \cdot Q + d_1 \cdot C_2 \stackrel{?}{=} b_1\]
4 Statement 1 commitment — \(G\)-component \[r_2 \cdot G + d_2 \cdot C_1 \stackrel{?}{=} a_2\]
5 Statement 1 commitment — \(Q\)-component \[r_2 \cdot Q + d_2 \cdot (C_2 - G) \stackrel{?}{=} b_2\]

Accept if and only if all five checks pass.

Security Properties of the 1-of-2 ZKP

Completeness: An honest prover with \(v \in \{0,1\}\) always satisfies all five verification checks.

Soundness: A cheating prover encrypting \(v \notin \{0,1\}\) cannot produce a valid proof. The probability of forging a valid proof is at most \(\frac{1}{|\mathbb{F}_n|} = \frac{1}{n}\), negligible in the security parameter.

Zero-Knowledge: The proof transcript \((a_1, b_1, a_2, b_2, d_1, d_2, r_1, r_2)\) is computationally indistinguishable from a simulated transcript that requires no knowledge of the vote \(v\) or the randomness \(r\). The verifier learns nothing about which of \(\{0, 1\}\) was encrypted.
